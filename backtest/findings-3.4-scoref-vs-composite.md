# 阶段 3.4 · scoreF 与 composite 的关系厘清

**时间**：2026-04-17
**数据**：V2 回测 · 45 月含熊市

---

## 🎯 核心结论：不是 bug，是有意设计

### 代码事实

| 评分 | 定义于 | 被使用于 | 目的 |
|---|---|---|---|
| **composite** | `market.js analyzeCategoryPerf` 内部 | `selectFunds`（**选基决策**） | 排序挑"最值得买" |
| **scoreF** | `score.js scoreF()` | 调仓建议、持仓诊断、UI 分数展示 | 评估"这基金多好"|

### 两套分数的算法差异

| 维度 | composite | scoreF |
|---|---|---|
| Calmar（风险调整收益） | **权重 ~80%** | 32% |
| 趋势一致性 | 20% | 24% |
| 任期稳定性 | 小 | 22% |
| 规模适配 | 无 | 10% |
| 费率 | 无 | 12% |
| 估值调整（PE 百分位） | 无 | ±10 |
| 信息比率 IR（alpha） | 无 | 隐含 |

**composite 更"粗"+重 calmar**，**scoreF 更"全"+多维度**。

## 🔬 对照实验：selectFunds 用 scoreF 替代 composite

跑 45 月含熊市数据，把 `f.composite = f.scoreF` 再跑 selectFunds：

| 画像 | composite（现状） | scoreF（实验） | Δ |
|---|---|---|---|
| 平衡+2年 夏普 | 0.89 | 0.67 | **-0.22** |
| 平衡+3年 夏普 | 0.89 | 0.75 | **-0.14** |
| 进取+2年 夏普 | 1.03 | 0.68 | **-0.35** |
| 进取+3年 夏普 | 1.10 | 0.82 | **-0.28** |
| 累计收益 Δ | — | **-8.69pp** | — |
| 最大回撤 Δ | — | **+7-11pp** | — |

**判定：🔴 21/21 画像全部恶化，平均夏普 -0.18**

**结论：scoreF 作为选基依据反而全面劣于 composite。**

---

## 💡 为什么 scoreF 选基更差？

**composite 是"纯 alpha 追踪器"**：
- 80% 权重在 Calmar（风险调整收益）
- 小量趋势动量
- 公式简单、噪声小

**scoreF 是"综合体检报告"**：
- Calmar 只占 32%
- 加入规模/费率/任期/估值 — 这些维度**稀释了"谁真的在跑赢"的信号**
- 在熊市中：
  - 规模大的基金反而调头慢（规模加分反作用）
  - 老经理可能过度保守（任期加分反作用）
  - 费率高低在几年维度影响微弱（费率加分无用）

**核心洞察**：选基需要的是"能跑赢的信号"，诊断需要的是"综合体检"。两者追求不同。

## ✅ 修订后的产品认知

### 之前的混乱
- "这基金 scoreF 85 分（展示），但智能方案没推荐它" → 用户困惑

### 正解
- **selectFunds** 优先挑 **composite** 高的基金（追 alpha）
- **scoreF** 是基金的"综合评分"（展示 + 诊断用）
- 两者有一定正相关（Spearman 0.7-0.9）但**边界不一致是正常的**

### 建议的 UI 改进（不必本轮做）
在"智能方案"页面把基金评分改为显示 composite 而非 scoreF（或同时显示两者），避免用户混淆。
但这动 UI 的改动量较大，优先级不高。

## 🎯 对 findings 的最终修订

**原 findings-v3 的疑问**：
> V2 发现 selectFunds 实际只用 composite 不用 scoreF，这是 bug 吗？

**修订答案（基于 3.4）**：
- **不是 bug**，两套分数服务不同目的
- 改用 scoreF 选基会让夏普降 0.18、累计降 8.7pp — 是明确的反向证据
- 现有设计是正确的，应该保留

## 📋 不做的事

1. **不改 selectFunds 的排序依据**（composite 是对的）
2. **不合并两套分数**（它们各司其职）
3. **不必在 UI 上做大改**（Spearman 相关 0.7+，用户不会看到强烈的不一致）
